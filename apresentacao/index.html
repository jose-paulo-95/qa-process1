Metodologia: Por que Kanban?

Kanban é uma metodologia ágil de gestão de trabalho que visualiza o fluxo de tarefas em um quadro, permitindo entrega contínua e melhoria constante através de limites de trabalho em progresso (WIP) e métricas de fluxo.

Vantagens

  • Flexibilidade para demandas variáveis
  • Foco em fluxo contínuo
  • Menos cerimônias, mais execução
  • Melhor para entrega contínua
  • Visualização clara do trabalho

Benefícios para QA

  • QA pega trabalho assim que está pronto
  • Testes acontecem continuamente
  • Fácil ver o que aguarda teste
  • WIP limits evitam sobrecarga
  • Métricas claras (lead time, cycle time)

Princípios do Kanban

1. Visualizar o trabalho | 2. Limitar WIP | 3. Gerenciar fluxo | 4. Políticas explícitas | 5. Melhorar colaborativamente

Definições de Pronto: DoR e DoD

Definido o modelo de processo que iremos adotar, agora precisamos estabelecer critérios claros para determinar quando uma demanda está pronta para início e quando ela está pronta para entrega.

Por que são importantes?

As definições de pronto garantem clareza e qualidade em cada etapa do processo, evitando retrabalho e bloqueios.

DoR vs DoD

DoR define quando uma story está pronta para ser trabalhada.
DoD define quando uma story está pronta para ser entregue.

Benefícios

✓ Reduz bloqueios durante desenvolvimento | ✓ Garante qualidade antes da entrega | ✓ Melhora comunicação entre equipes | ✓ Acelera o fluxo de trabalho

9 / 39

Definition of Ready (DoR)

Uma story está PRONTA PARA SER TRABALHADA quando:

  • ✓ Requisitos claros e documentados
  • ✓ Critérios de aceite definidos
  • ✓ Story estimada pela equipe
  • ✓ Dependências identificadas e resolvidas
  • ✓ Design/UX aprovado (quando aplicável)
  • ✓ Ambiente de desenvolvimento disponível
  • ✓ Acesso necessário concedido
  • ✓ Dados de teste definidos
  • ✓ Priorizada pelo Product Owner
  • ✓ Aprovada para entrar no fluxo

Objetivo do DoR

Garantir que stories no backlog estão completas e prontas para desenvolvimento, evitando bloqueios e retrabalho durante a implementação.

10 / 39

Definição de Pronto (DoD)

Uma feature/story é considerada PRONTA quando:

  • ✓ Código desenvolvido e revisado
  • ✓ Testes unitários criados e passando
  • ✓ Testes funcionais executados
  • ✓ Testes de regressão executados
  • ✓ Bugs críticos e altos resolvidos
  • ✓ Documentação atualizada
  • ✓ Deploy em ambiente de QA/Staging
  • ✓ Aprovado pelo QA
  • ✓ Aprovado pelo Product Owner
  • ✓ Pendências técnicas resolvidas

Definição de Pronto para Teste

Story pronta para QA quando: código desenvolvido, deploy em teste, ambiente disponível, dados preparados

Processo de QA: Garantia de Qualidade

O que é o Processo de QA?

Conjunto estruturado de atividades que garantem a qualidade do software através de planejamento, execução e análise de testes.

Por que é essencial?

Identifica defeitos antes da produção, valida requisitos, garante experiência do usuário e reduz custos de correção.

Componentes do Processo

✓ Ciclo de vida estruturado | ✓ Diferentes tipos de teste | ✓ Templates padronizados | ✓ Métricas e acompanhamento

Integração BrowserStack Test Management

Por que BrowserStack?

✓ Redução de custos (armazenamento incluído no plano)
✓ Execução integrada de testes
✓ Rastreabilidade completa com Jira

Configuração

  • Campo "BrowserStack Test Case URL"
  • Campo "BrowserStack Test Case ID"
  • Link direto entre Jira e BrowserStack
  • Rastreabilidade bidirecional

Workflow

  1. Criar Test Case no Jira
  2. Criar caso no BrowserStack
  3. Fazer link via URL/ID
  4. Manter sincronizado

Perguntas Frequentes

Quanto tempo leva para ver resultados?

Resultados iniciais em 2-4 semanas. Melhorias significativas em 2-3 meses.

Precisamos de ferramentas adicionais?

Não. Usamos Jira (já possui) e BrowserStack (já possui). Processo pode ser implementado com ferramentas existentes.

E se a equipe resistir à mudança?

Processo inclui treinamento, comunicação clara e envolvimento da equipe. Mudanças são graduais e baseadas em feedback.

Como garantimos que o processo será seguido?

Métricas de acompanhamento, revisões regulares, liderança comprometida e cultura de melhoria contínua.
